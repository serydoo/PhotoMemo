# Fixture Specification

Last updated: 2026-06-20

## Purpose

MemoMark needs a repeatable fixture library for long-term regression verification.

This fixture set is intended to support:

- metadata read verification
- metadata normalization verification
- variable resolution verification
- export verification
- export read-back verification
- batch verification

## Repository Policy

MemoMark does **not** commit copyrighted personal photos.

Current repository policy:

- keep the fixture directory structure in source control
- keep a manifest and naming contract in source control
- add real fixture binaries only when they are either:
  - project-owned synthetic samples, or
  - explicitly licensed for repository inclusion

## Required Characteristics

### 01_iPhone_JPEG.jpg

- JPEG
- Apple-origin still photo
- has EXIF capture date
- has TIFF make/model
- has typical iPhone camera fields

### 02_iPhone_HEIC.heic

- HEIC
- Apple-origin still photo
- has EXIF capture date
- has TIFF make/model
- used for HEIC metadata retention comparison

### 03_Canon.CR3

- optional placeholder for future RAW-read policy decisions
- not required for the first automated test batch
- should remain documentation-only until RAW support direction is explicit

### 04_Nikon.JPG

- non-Apple JPEG
- has EXIF + TIFF camera metadata
- should differ from iPhone naming and lens metadata conventions

### 05_GPS.jpg

- includes valid GPS latitude/longitude
- ideally also altitude
- preferred source for location normalization verification

### 06_NoGPS.jpg

- similar capture scenario without GPS
- verifies graceful degradation

### 07_Portrait.jpg

- width < height
- used for orientation/aspect-ratio coverage

### 08_Landscape.jpg

- width > height
- used for orientation/aspect-ratio coverage

### 09_LivePhotoStill.heic

- still resource exported from a Live Photo
- used only for boundary verification
- does not imply full Live Photo support

### 10_LowMetadata.jpg

- minimal metadata case
- missing most optional fields
- should still import and degrade gracefully

## Naming Rules

- keep a two-digit numeric prefix for stable ordering
- use the original file extension when known
- avoid spaces in filenames
- avoid personal names, locations, or private semantic labels
- if a fixture changes meaning, create a new number instead of silently replacing behavior

## Fixture Acceptance Rules

A fixture is acceptable only if:

- it has documented provenance
- its expected metadata characteristics are written down
- its intended test coverage is mapped in `Docs/RegressionMatrix.md`

## Synthetic Regeneration

Committed synthetic fixtures can be regenerated with:

```bash
xcrun swift /Users/rui/Desktop/PhotoMemo/Tests/Fixtures/GenerateSyntheticFixtures.swift
```

Regeneration is expected to be deterministic.

## Future Synthetic Fixture Strategy

Preferred future additions:

- generated JPEG/PNG fixtures with controlled dimensions
- project-owned still images with explicitly injected metadata
- metadata-only edge-case samples created for regression protection

This strategy is now active for the first committed fixture-backed export tests.
