# 项目架构（技术角度）

## 总体判断

PhotoMemo 当前不是一个“单体 View 工程”，而是一个正在从 V1 产品代码逐步走向 V2 架构边界的并存型仓库。

最重要的技术事实有两个：

1. V1 仍然是真实可读、可扩展的主代码入口之一
2. V2 的架构边界已经开始通过 `ConfigurationCenter`、`MemoryEngine`、`ConfigurationSnapshot` 等对象向前推进

## 当前 iOS 入口链路

实际入口不是直接进 V1，而是：

```text
PhotoMemoiOSApp / PhotoMemoiOSV1App
-> PhotoMemoiOSHomeView
-> PhotoMemoRootSceneView
-> PhotoMemoiOSTemporaryEntryView
-> PhotoMemoiOSV1View 或 ConfigurationCenteriOSView
```

## 当前仓库最重要的技术现实

### 并存，而不是替换

当前不是“V1 已死，V2 已完全接管”的状态。

而是：

- V1 页面仍然存在
- Configuration Center 已经形成新的对象结构
- Memory Engine 已经开始进入生产路径
- 共享运行时和共享服务在给这两边供血

### 环境装配是一个中心节点

[AppEnvironment.swift](/Users/rui/Desktop/PhotoMemo-main-archive-20260702/Source/PhotoMemo/PhotoMemo/Architecture/AppEnvironment.swift) 是理解整个 app 的一个关键入口。

它集中装配：

- services
- repositories
- coordinators
- queue store
- external intake center

这意味着它非常适合做“系统装配入口图”的中心节点，但也意味着它是一个不要轻易乱拆的文件。
