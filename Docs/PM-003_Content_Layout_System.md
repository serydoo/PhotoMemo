## PM-003 Goal

PM-003 defines the content semantics that the future content and layout system must preserve.

PhotoMemo no longer treats the bottom information area as an EXIF border or metadata dump.

PhotoMemo treats it as a memory expression system.

The system must answer four different questions:

- who recorded this moment
- how this moment was recorded
- when this moment was recorded
- where this moment belongs in a life timeline

## Frozen Decision: Semantic Slot Principle

Slot always represents:

```text
semantic meaning
```

Slot never represents:

```text
layout position
```

The future Layout Engine may decide visual arrangement.

PM-003 defines what each slot means before any future layout work.

## Frozen Semantic Slots

### Slot A

Name:

```text
Recorder
```

Question it answers:

```text
谁记录了这一刻？
```

Meaning:

- Slot A expresses the recorder identity
- Slot A belongs to memory expression, not camera metadata
- Slot A may draw from profile or recorder-oriented variables

### Slot B

Name:

```text
Capture Summary
```

Question it answers:

```text
这一刻是怎样记录下来的？
```

Meaning:

- Slot B is not a Metadata Viewer
- Slot B is not an EXIF Viewer
- Slot B only expresses the photographic summary of the capture
- Slot B may use selected capture-oriented variables, but only as summarized expression

### Slot C

Name:

```text
Timeline
```

Question it answers:

```text
这一刻记录于什么时候？
```

Default expression:

```text
记录于｜日期｜时间
```

Frozen rules:

- Timeline Action default is `记录于`
- seconds do not display
- Slot C expresses record time, not parameter density

### Slot D

Name:

```text
Time Anchor
```

Question it answers:

```text
这一刻位于人生哪个重要事件？
```

Frozen rules:

- Slot D does not display Metadata
- Slot D only displays Life Anchor Expression
- Slot D is the semantic bridge between a photo timestamp and a life event

## Frozen Decision: Time Anchor System

Time Anchor is not defined as a raw date.

Time Anchor is defined as a:

```text
Life Event
```

Examples:

- 出生
- 结婚
- 高考
- 退休
- 大学毕业
- 第一次旅行
- any meaningful custom life event

Date is only an attribute of the Life Event.

Life Event is the true anchor.

## Frozen Decision: Life Anchor V1

V1 supports only:

```text
3 user-defined Life Anchors
```

Each Life Anchor includes these fields at the architecture level:

- `name`
- `date`
- `description`
- `category` reserved
- `enabled` reserved

V1 implementation scope is frozen to:

- `name`
- `date`
- `description`

Reserved only in V1:

- `category`
- `enabled`

This preserves future architecture without expanding current runtime scope.

## Frozen Decision: Time Anchor Principle

PhotoMemo does not directly display time.

PhotoMemo displays:

```text
the distance between a person and an important life event
```

Time Anchor supports:

- past
- future

Both directions are unified by the same calculation model.

Examples:

- past: `宝宝今天 2岁3个月18天`
- future: `距离高考还有 6210 天`

Unified ownership:

```text
Time Anchor Engine
```

## Frozen Decision: Slot D Grammar

Time Anchor Expression uses a fixed four-part grammar:

```text
Subject
-> Anchor Prompt
-> Anchor Result
-> Anchor Suffix
```

### Subject

Source:

- comes from Profile

Rules:

- editable
- human-facing
- not generated by the calculation engine

### Anchor Prompt

Source:

- user-authored expression text

Rules:

- editable
- expresses relationship framing

Examples:

- `今天`
- `距离出生`
- `陪伴我们`
- `已经`
- `来到这个世界`

### Anchor Result

Source:

- calculated by Time Anchor Engine

Rules:

- auto-generated
- not editable
- never hard-coded into expression text

### Anchor Suffix

Source:

- user-authored ending text

Rules:

- editable
- closes the expression tone

Examples:

- `啦！`
- `。`
- `❤️`

## Frozen Decision: Expression Principle

Expression is responsible for:

- wording
- sentence tone
- presentation framing

Engine is responsible for:

- time-distance calculation
- past/future evaluation
- reusable result generation

Expression and Engine must remain fully decoupled.

PhotoMemo must not directly concatenate a full sentence inside the calculation layer.

PhotoMemo must use Grammar as the contract between expression and calculation.

## Frozen Decision: Variable Classification

The variable library must be reorganized around semantic ownership.

PM-003 frozen variable categories are:

- Recorder
- Capture
- Timeline
- Time Anchor
- Memory
- Profile
- Metadata
- Custom
- Action
- Timeline Action

Meaning:

- variables are grouped by semantic role
- variable categories must serve expression clarity
- variable categories must not collapse back into a generic EXIF list

## Frozen Decision: Typography Strategy

Typography in PM-003 is frozen at the semantic-strategy level.

Frozen rules:

- typography serves semantic hierarchy
- typography follows slot meaning, not metadata density
- Recorder, Capture Summary, Timeline, and Time Anchor are distinct expression groups
- Time Anchor remains an expression block, not a parameter block

Detailed measurable typography values remain future Layout Specification work.

PM-003 only freezes semantic ownership and hierarchy.

## Architecture Impact

PM-003 clarifies the future ownership chain:

```text
Profile / Metadata / Life Anchor
-> Time Anchor Engine
-> Presentation Grammar
-> Semantic Slots
-> Layout Engine
-> Renderer
```

This means:

- Life Anchor semantics belong to the memory and expression architecture
- Slot definitions belong to content architecture
- layout still remains a later concern
- renderer remains downstream implementation only

## Frozen Summary

PM-003 Phase 1 freezes:

- Semantic Slot Principle
- Slot A = Recorder
- Slot B = Capture Summary
- Slot C = Timeline
- Slot D = Time Anchor
- Life Anchor as Life Event, not raw Date
- Life Anchor V1 field boundary
- Time Anchor past/future unified calculation principle
- Slot D four-part grammar
- Expression and Engine decoupling
- semantic variable categories
- typography strategy at the semantic level
